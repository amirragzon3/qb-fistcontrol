# Contributing to QB-FistControl

Thank you for your interest in contributing to QB-FistControl! This document provides guidelines and instructions for contributing to the project.

## 🤝 Ways to Contribute

- **Bug Reports**: Report bugs you find
- **Feature Requests**: Suggest new features
- **Code Contributions**: Submit pull requests with improvements
- **Documentation**: Improve documentation and README
- **Translations**: Help translate to other languages

## 🐛 Reporting Bugs

Before creating a bug report, check the [issue list](https://github.com/amirragzon3/qb-fistcontrol/issues) to avoid duplicates.

### Bug Report Template
```
**Description**: Brief description of the bug
**Steps to Reproduce**: 
1. Step 1
2. Step 2
**Expected Behavior**: What should happen
**Actual Behavior**: What actually happens
**Server Config**: Your server.cfg (if relevant)
**Error Logs**: Any console errors
```

## 💡 Feature Requests

Describe the feature you'd like to see and why it would be useful.

### Feature Request Template
```
**Description**: What feature would you like?
**Use Case**: Why do you need this?
**Example**: How would you use it?
```

## 🔧 Code Contributions

### Getting Started

1. **Fork** the repository
2. **Clone** your fork
   ```bash
   git clone https://github.com/YOUR-USERNAME/qb-fistcontrol.git
   ```
3. **Create a branch** for your feature
   ```bash
   git checkout -b feature/your-feature-name
   ```
4. **Make your changes**
5. **Commit** with clear messages
   ```bash
   git commit -m "Add: description of your changes"
   ```
6. **Push** to your fork
   ```bash
   git push origin feature/your-feature-name
   ```
7. **Create a Pull Request**

### Commit Message Format

Follow this format for commit messages:

```
<type>: <subject>

<body>

<footer>
```

**Types**:
- `feat`: A new feature
- `fix`: A bug fix
- `docs`: Documentation changes
- `style`: Code style changes (formatting, etc.)
- `refactor`: Code refactoring
- `perf`: Performance improvements
- `test`: Adding or updating tests
- `chore`: Maintenance tasks

**Examples**:
```
feat: Add custom punch animations
fix: Resolve Right-Click detection issue
docs: Update installation guide
```

### Code Style

- Use **Lua** 5.1 style
- Add comments for complex logic
- Use **lowercase** with underscores for variables
- Use **UPPERCASE** for constants
- Keep lines under 100 characters

### Testing

Before submitting a PR:
1. Test locally on a FiveM server
2. Verify the feature works as intended
3. Check for performance impacts
4. Ensure backwards compatibility

## 📋 Pull Request Process

1. Update the README.md if changing functionality
2. Update CHANGELOG.md with your changes
3. Ensure your code follows the style guide
4. Provide a clear description of the changes
5. Link any related issues
6. Wait for review and feedback

### PR Template

```markdown
## Description
Brief description of your changes

## Type of Change
- [ ] Bug fix
- [ ] New feature
- [ ] Breaking change
- [ ] Documentation update

## Testing
Describe how you tested this

## Checklist
- [ ] Code follows style guide
- [ ] README updated
- [ ] CHANGELOG updated
- [ ] Tested on local server
```
